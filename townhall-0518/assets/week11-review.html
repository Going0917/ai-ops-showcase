Part 7：本周认知收获

1. "lisa 沟通诺特复计"——客户问题的"行业唯一性"判断改变解法

诺特兰德的归因复计问题（小店直购全店归因 + 同店 ADQ + 达人分佣 + 微信豆四源叠加 → 50万/天 GAP）困扰了从 4/27 到 5/12 整整 2 周。5/12 lisa 沟通后明确：消费品行业仅诺特存在该问题——其他客户要么没有四源叠加，要么客单价/品类结构不引发同等量级 GAP。

核心洞察：客户问题的"行业唯一性"判断决定解法路径——

  • 如果是行业普遍问题 → 推动产研改归因口径（耗时长、ROI 低、可能影响他人）
  • 如果是客户独有问题 → 推动客户改投放结构（"开新店做归因隔离"，立竿见影）

这个方法论值得固化到 ad-review-expert SKILL 的"客户问题判断 SOP"中——先问行业唯一性，再决定推产研还是推客户。

2. "21:00 加班补天"——单日产能从 5 项 → 12 项的跃迁机制

5/14 全天本来已经完成 5 项 P0，21:00 自动 Mode J 归档完成后，继续工作 2.5 小时连续完成 4 大方向交付——OnePage 双版 + 双周会 docx 三轮 + Townhall 大纲实质修订 + wecom-doc bug 修复。

23:39 用户主动反馈"今天实在干了太多活"，触发 Mode J 二次补记，最终单日交付物从 5 项 → 12 项。

核心洞察：晚间是"高密度低门槛"产能窗口——白天有会议、客户、协作打断，晚间是连续闭环 + 工程化收尾的最佳窗口。但代价是 Mode L 兜底机制失效（21:00-23:39 间无任何实时记录），全部依赖反推。

3. "校验 J-4"沉淀——日期+工作日+DDL 三重校验是 AI 协作的基础健全性

5/15 晚归档反推时，AI 把"5/16 周六"排成"双周会 DDL + 4 项 P0 必推"——把双休强行排成工作日。用户反馈"你的设计好离谱呀，我周六周日是双休的呀"后，沉淀「校验 J-4：日期+工作日+DDL 三重校验」。

核心洞察：AI 协作的"基础健全性"问题往往最致命——比起复杂的方法论错误，更易出错的是"周几对应日期 / 自动化调度范围 / DDL 与工作日的关系"这类基础认知。

4. "纯能力建设日"vs "P0 推进"——5/15 暴露的资源分配失衡

5/15 是"纯能力建设日"——commit 7a7efd1 共 +6396 行。但对 P0 推进的牺牲过大：4 项 P0 全天未推进顺延 W12。

核心洞察："大块能力建设日"应预留至少 1 小时给 P0 紧急事项。下次类似场景，建议早晨 9:00-10:00 优先处理 P0 + 其他时间投入能力建设。这个原则值得固化到 Mode I 晨间规划——P0 事项保护时段（9:00-10:00 + 17:00-18:00）不应被能力建设占用。

⚠️ 待改进：Mode L 实时积累机制覆盖率不足（晚间加班场景）

5/14 21:00-23:39 加班期间无任何 Mode L 实时记录，5/15 14:00 后产出（双周会执行 + SKILL 升级 + Townhall deck）也无 Mode L 实时记录。Mode J 反推时晚间加班场景的反推质量明显低于白天。

改进方向：

  1. 短期：每个有形交付物落地时，强制 AI 主动 append 一条简短 Mode L 条目
  2. 中期：Mode L 触发条件扩展——除"有形交付物"外，"用户反馈/校准/方向变更"也强制写入
  3. 长期：Mode L 触发频率从"事件驱动"升级为"事件+时段双驱动"

⚠️ 待改进：自动化故障检测时滞过长（13.5h 盲区）

5/15 早 8:30 Mode I 未触发，但用户 21:56 才发现；5/15 晚 21:00 Mode J 未触发，用户 22:24 才发现。故障识别延迟最长达 13.5 小时。

改进方向：

  1. 晨间自检：Mode I 触发后第一步立即写入"心跳文件"
  2. 前一晚兜底：Mode J 步骤 1（日志检查）增加"昨日 morning 文件存在性自检"
  3. 早间用户提醒：Mode I 触发后 9:00 自动发送企微消息提醒用户

5/15 全天 0 项 KR1-3 对客交付，4 项 P0 全天未推进。这是连续第二次出现"大块能力建设日挤压 P0"的模式（4/24 也是同样问题）。
